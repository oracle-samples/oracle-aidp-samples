# ADW External Table on Delta UniForm

This sample demonstrates how to:

1. Create an ADW stored procedure that locates the latest Iceberg-compatible `metadata/vN.metadata.json` generated by a Delta table with **UniForm enabled for Iceberg**.
2. Call that procedure from Python in a notebook after writing data to the Delta table.
3. Recreate an ADW external table against the latest metadata file so ADW can query the table.

The sample artifacts are:

- [refresh_iceberg_external_table.sql](./refresh_iceberg_external_table.sql)
- [table_generation.ipynb](./table_generation.ipynb)

## Why this sample exists

ADW direct metadata access for Iceberg is a **point-in-time** pattern. If the table generates a newer `vN.metadata.json`, ADW does not automatically follow it unless you either:

- use a supported catalog-managed integration, or
- recreate the external table against the latest metadata file

This sample automates the second approach for a Delta table with UniForm enabled for Iceberg.

Relevant references:

- Oracle ADW Iceberg external tables:
  https://docs.oracle.com/en-us/iaas/autonomous-database-serverless/doc/query-external-data-apache-iceberg.html
- Delta UniForm:
  https://docs.delta.io/delta-uniform/

## Tested environment

This sample has been tested with:

- AIDP Workbench for the Spark / Delta UniForm side
- custom Spark based on Spark 3.5.0
- Delta Lake 3.2.0
- ADW 26ai for the external Iceberg table side

## Execution order

Run the sample in this order:

1. Run the SQL script in ADW.
2. Run the notebook in your Spark environment.

Do not start with the notebook. The notebook assumes the ADW procedure already exists.

## Step 1: Run the SQL script in ADW

Execute [refresh_iceberg_external_table.sql](./refresh_iceberg_external_table.sql) in ADW.

This script does three things:

1. Creates or recreates the `DBMS_CLOUD` credential used to access OCI Object Storage.
2. Grants network ACL access to `objectstorage.<region>.oraclecloud.com`.
3. Creates the `refresh_iceberg_external_table` procedure.

### Prerequisites before running the SQL

Before running the SQL script, make sure you have:

- an ADW schema where you can create procedures
- privilege to run `DBMS_CLOUD.CREATE_CREDENTIAL`
- privilege to run `DBMS_NETWORK_ACL_ADMIN.APPEND_HOST_ACE`
- privilege to create or replace the procedure
- an OCI API signing key pair for Object Storage access
- the OCI identifiers needed to create the ADW credential:
  - user OCID
  - tenancy OCID
  - private key
  - fingerprint

Operationally, the SQL script assumes the executing ADW schema can:

- create the Object Storage credential
- grant or append the required outbound ACL to Object Storage
- create or replace the `refresh_iceberg_external_table` procedure

### Inputs to customize in the SQL file

Replace the placeholders in the SQL file before running it:

- `<name_of_credential>`
- `ocid1.user.oc1...`
- `ocid1.tenancy.oc1...`
- private key
- fingerprint
- `<region>`
- `<principal_name>`

You should also review the example procedure call at the bottom of the SQL file if you plan to use it for manual testing.

### ACL nuance

The ACL grant must be given to the ADW schema that executes `DBMS_CLOUD` operations. In many demos that is `ADMIN`, but use the real schema for your setup.

The SQL file also includes a verification query against `USER_HOST_ACES`.

If the refresh procedure later fails with an error that does not clearly point at the cause, see [Manually verifying credentials and external table creation](#manually-verifying-credentials-and-external-table-creation) to run the underlying steps by hand and isolate the failing layer.

## Step 2: Run the notebook

Open [table_generation.ipynb](./table_generation.ipynb) and run it top to bottom.

The notebook:

1. Creates a Delta table with UniForm enabled for Iceberg.
2. Writes sample data to the table.
3. Reads the Delta table location with `DESCRIBE DETAIL`.
4. Connects to ADW with `python-oracledb`.
5. Calls the ADW procedure with:
   - ADW table name
   - ADW credential name
   - OCI region
   - the base `oci://bucket@namespace/path/to/table` location

The procedure then:

1. Converts the OCI URI to the HTTPS Object Storage base path expected by ADW.
2. Lists the metadata directory in Object Storage.
3. Picks the highest `vN.metadata.json`.
4. Drops the ADW external table if it exists.
5. Recreates the ADW external table against that metadata file.

When the notebook creates the Delta table, it explicitly sets both documented UniForm Iceberg properties so the sample remains reproducible on Delta 3.2.0:

- `delta.enableIcebergCompatV2 = true`
- `delta.universalFormat.enabledFormats = iceberg`

The procedure uses `DBMS_ASSERT.SIMPLE_SQL_NAME` when dropping the ADW table before recreation. Because of that, `p_table_name` must be a simple ADW table name such as `UNIFORM_TABLE_01`, not a qualified name such as `ADMIN.UNIFORM_TABLE_01`.

In this sample, the ADW schema is derived from the Python connection user. The procedure is created in that schema, and the recreated external table is also created in that same connected schema.

The procedure intentionally scans the `metadata/` directory for the highest `vN.metadata.json` instead of relying on `version-hint.text`.

`version-hint.text` is a useful convenience pointer, but the versioned metadata files are the stronger source of truth. Scanning the available `vN.metadata.json` files is more robust in cases where the hint file is missing, stale, or not updated exactly when the latest metadata commit becomes visible.

Also note that Iceberg-compatible metadata generation for Delta UniForm can be asynchronous. Because of that, there can be a short delay between a Delta write and the appearance of the newest `vN.metadata.json` under the `metadata/` directory. If the latest commit is not yet reflected in Object Storage, wait briefly and rerun the refresh procedure.

When changes are made to the Delta UniForm table, such as:

- new data being written
- additional Delta commits being created
- schema evolution reflected in newer Iceberg-compatible metadata

rerunning the refresh procedure updates the ADW external table to point at the latest available `vN.metadata.json`, bringing the ADW view back in sync with the latest published Iceberg-compatible metadata.

### Caveat: brief drop and recreate window

The refresh procedure works by dropping and recreating the ADW external table.

That means there is a short interval during refresh when the ADW table does not exist. During that interval, concurrent access may fail or briefly see a glitch.

For a demo this is usually acceptable, but for production-style usage you should be aware of this drop/recreate window and plan around it if the table is queried concurrently.

## mTLS and wallet requirements

The notebook assumes **mTLS** is required for the ADW connection.

Important nuances:

- If the ADB page shows **Mutual TLS**, the Python connection needs the wallet.
- For `python-oracledb` Thin mode, the wallet directory typically needs:
  - `tnsnames.ora`
  - `ewallet.pem`
- `config_dir` points to `tnsnames.ora`.
- `wallet_location` points to `ewallet.pem`.
- In most setups, `config_dir` and `wallet_location` are the same directory.
- `wallet_password` is the wallet zip download password, not the ADW schema password.

If your ADW allows walletless TLS, you can simplify the Python connection, but this sample keeps the mTLS path because it is the stricter and more common ADB setup.

## Partitioning note

The sample uses a minimal non-partitioned Delta table, but the same approach works for **partitioned** Delta tables with UniForm enabled.

ADW queries the table through the Iceberg-compatible metadata, so the presence of partition directories in the underlying object storage layout does not change the overall integration pattern.

You can extend the notebook to use a `PARTITIONED BY (...)` table definition and the ADW-side procedure still follows the latest Iceberg metadata file in the same way.

## Existing Delta tables and REORG

This sample shows a simple table created with UniForm enabled from the beginning. For an existing Delta table, the enablement flow can be different.

In that case, you may need to:

1. run a `REORG TABLE ... APPLY (UPGRADE UNIFORM(...))` step
2. alter the table properties to enable the required Iceberg-compatible UniForm settings
3. perform a subsequent Delta write so Iceberg-compatible metadata is emitted under the `metadata/` directory

A generic example looks like this:

```python
spark.sql(
    f"""REORG TABLE {table_name}
    APPLY (UPGRADE UNIFORM(ICEBERG_COMPAT_VERSION=2))"""
)

spark.sql(
    f"""ALTER TABLE {table_name}
    SET TBLPROPERTIES(
      'delta.enableIcebergCompatV2' = 'true',
      'delta.universalFormat.enabledFormats' = 'iceberg'
    )"""
)

# A later Delta commit will cause the Iceberg-compatible metadata directory to appear or advance.
spark.sql(f"INSERT INTO {table_name} VALUES (...) ")
```

In practice, one important nuance is that the `metadata/` directory may not appear immediately after the `REORG` and `ALTER TABLE` statements alone. A later write or commit may be needed before the Iceberg-compatible metadata becomes visible.

Operational caveat:

- `REORG` can rewrite data files
- it can consume additional compute and I/O
- it can take noticeable time on larger tables
- it can temporarily increase storage usage during the rewrite

So while `REORG` is useful for upgrading existing tables, it should be treated as an operational change and scheduled with care for larger or actively used datasets.

## Manually verifying credentials and external table creation

The refresh procedure bundles several steps — credential access, the Object Storage ACL, metadata discovery, and the external table DDL — into a single call. When it fails, ADW sometimes returns an error that does not clearly point at the real cause. In practice the underlying problem is almost always one of:

- a `DBMS_CLOUD` credential that is missing, wrong, or built from a mismatched key/fingerprint
- the Object Storage host ACL not granted to the connected schema
- the latest `vN.metadata.json` not yet visible, because UniForm Iceberg metadata generation is asynchronous

Running the same steps by hand isolates which layer is failing, because you see the raw error at each step instead of the procedure's summarized one. Run the statements below while connected as the **same ADW schema the notebook uses**, and replace every `<...>` placeholder with your values.

> Work through the steps in order. The first one that fails points at the layer to fix. The `DROP` wrappers in steps 1, 4, and 7 deliberately ignore errors from dropping an object that does not exist so the blocks can be re-run; they are conveniences, not diagnostic steps — the `CREATE_CREDENTIAL`, `LIST_OBJECTS`, `CREATE_EXTERNAL_TABLE`, and `SELECT` calls are where a real error surfaces.

### 1. Create (or recreate) the credential

```sql
BEGIN
   BEGIN
      DBMS_CLOUD.DROP_CREDENTIAL('<CREDENTIAL_NAME>');
   EXCEPTION
      WHEN OTHERS THEN
         -- Best-effort drop so this block can be re-run. This swallows any
         -- error here (usually the credential not existing yet); the
         -- CREATE_CREDENTIAL below is the step that must actually succeed.
         NULL;
   END;

   DBMS_CLOUD.CREATE_CREDENTIAL(
      credential_name => '<CREDENTIAL_NAME>',
      user_ocid       => 'ocid1.user.oc1...',
      tenancy_ocid    => 'ocid1.tenancy.oc1...',
      private_key     => '<private_key_content>',
      fingerprint     => '<fingerprint>'
   );
END;
/
```

`private_key` is the PEM body of your OCI API signing key, and `fingerprint` must match the public key uploaded to that user in OCI. A mismatch here is a common source of the misleading errors above — `CREATE_CREDENTIAL` itself succeeds, but every later call that uses the credential fails.

### 2. Grant and verify the Object Storage ACL

Grant outbound access to the Object Storage endpoint for your region. `principal_name` must be the schema that runs the `DBMS_CLOUD` calls (often `ADMIN`, but use your real schema).

```sql
BEGIN
  DBMS_NETWORK_ACL_ADMIN.APPEND_HOST_ACE(
    host => 'objectstorage.<region>.oraclecloud.com',
    ace  => xs$ace_type(
      privilege_list => xs$name_list('connect', 'resolve'),
      principal_name => '<principal_name>',
      principal_type => xs_acl.ptype_db
    )
  );
END;
/
```

Verify the grant. `USER_HOST_ACES` shows the ACEs that apply to the **currently connected** user, so run this as the schema you granted the ACL to:

```sql
SELECT host, lower_port, upper_port, privilege, status
FROM user_host_aces
WHERE host = 'objectstorage.<region>.oraclecloud.com';
```

Expected — two rows, both `GRANTED`:

```
objectstorage.<region>.oraclecloud.com    RESOLVE    GRANTED
objectstorage.<region>.oraclecloud.com    CONNECT    GRANTED
```

If you see no rows, either the ACL was not granted or it was granted to a different principal than the schema you are connected as.

### 3. Confirm the credential can reach Object Storage and find the metadata

This is the single most useful check: it exercises the credential and the ACL together, and it tells you whether the latest metadata file is actually visible yet. Point `location_uri` at the table's `metadata/` folder, using the HTTPS Object Storage form of your `oci://` path.

```sql
SELECT object_name,
       TO_NUMBER(
          REGEXP_SUBSTR(object_name, '^v([0-9]+)\.metadata\.json$', 1, 1, NULL, 1)
       ) AS version_no
FROM TABLE(
   DBMS_CLOUD.LIST_OBJECTS(
      credential_name => '<CREDENTIAL_NAME>',
      location_uri    => 'https://objectstorage.<region>.oraclecloud.com/n/<namespace>/b/<bucket>/o/<table_path>/metadata/'
   )
)
WHERE REGEXP_LIKE(object_name, '^v[0-9]+\.metadata\.json$')
ORDER BY version_no DESC;
```

Interpret the result:

- **An authorization / signature error** → the credential is wrong or the ACL is not granted to this schema. Fix steps 1–2.
- **No rows** → no `vN.metadata.json` was found at that path. Either the `location_uri` points at the wrong namespace, bucket, or table path (recheck the `oci://` → HTTPS mapping below), or the credential works but the metadata is not visible yet — UniForm Iceberg metadata generation is asynchronous, so trigger a Delta write/commit, wait briefly, and rerun.
- **One or more rows** → the first row (highest `version_no`) is the latest metadata file; that is the one the external table should point at. Ordering by the numeric `version_no` matters, because a plain string sort would place `v10.metadata.json` before `v9.metadata.json`.

The `oci://<bucket>@<namespace>/<table_path>` URI maps to `https://objectstorage.<region>.oraclecloud.com/n/<namespace>/b/<bucket>/o/<table_path>/` — the same conversion the procedure performs internally.

### 4. Drop the table if it already exists

Wrapped so it does not fail when the table is absent (`ORA-00942`):

```sql
BEGIN
   EXECUTE IMMEDIATE 'DROP TABLE <TABLE_NAME>';
EXCEPTION
   WHEN OTHERS THEN
      IF SQLCODE != -942 THEN
         RAISE;
      END IF;
END;
/
```

### 5. Create the external table against the metadata file

Use the full HTTPS URL of the highest `vN.metadata.json` you found in step 3:

```sql
BEGIN
   DBMS_CLOUD.CREATE_EXTERNAL_TABLE(
      table_name      => '<TABLE_NAME>',
      credential_name => '<CREDENTIAL_NAME>',
      file_uri_list   => 'https://objectstorage.<region>.oraclecloud.com/n/<namespace>/b/<bucket>/o/<table_path>/metadata/v<N>.metadata.json',
      format          => '{"access_protocol":{"protocol_type":"iceberg"}}'
   );
END;
/
```

If your `<table_path>` contains characters that require URL encoding (for example spaces or `@`), percent-encode them in the `file_uri_list`. The procedure does this automatically with `UTL_URL.ESCAPE`.

If your table has long string columns, the `maxvarchar` format option controls the width of the derived `VARCHAR2` columns. It accepts `standard` (`VARCHAR2(4000)`), `extended` (`VARCHAR2(32767)`, allowed only on an instance with `MAX_STRING_SIZE=EXTENDED`), and `auto` (the default — picks the width from the instance's `MAX_STRING_SIZE`). Autonomous Database defaults to `MAX_STRING_SIZE=EXTENDED`, so with `auto` you get `VARCHAR2(32767)` and no truncation. You only need to set `maxvarchar` explicitly if the derived columns come out as `VARCHAR2(4000)` — for example under `MAX_STRING_SIZE=STANDARD` or if you pass `"maxvarchar":"standard"` — in which case string values longer than 4000 bytes are returned as `NULL` (a conversion error, not a rejected row).

### 6. Query the table

```sql
SELECT * FROM <TABLE_NAME>;
```

If this returns rows, the end-to-end path — credential, ACL, metadata discovery, and external table — is working, and the procedure should succeed with the same inputs.

### 7. Clean up

Both drops are wrapped so cleanup does not fail if the table or credential is already gone:

```sql
BEGIN
   BEGIN
      EXECUTE IMMEDIATE 'DROP TABLE <TABLE_NAME>';
   EXCEPTION
      WHEN OTHERS THEN
         IF SQLCODE != -942 THEN
            RAISE;
         END IF;
   END;

   BEGIN
      DBMS_CLOUD.DROP_CREDENTIAL('<CREDENTIAL_NAME>');
   EXCEPTION
      WHEN OTHERS THEN
         -- Best-effort cleanup; swallows any error (usually the credential
         -- already being gone).
         NULL;
   END;
END;
/
```
